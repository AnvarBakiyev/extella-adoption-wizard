# Changelog

## 5.24 — 2026-07-22

- Клиентская QA-дельта больше не пытается публиковать глобальные системные эксперты токеном
  тестового Mac: такой токен вправе читать и запускать их, но закономерно получает HTTP 401 на
  `/api/expert/save`. Публикация отделена как операция владельца релиза.
- До любого копирования файлов обновлятор параллельно и только на чтение сверяет три обязательных
  эксперта с исходниками QA-релиза. Отсутствующая или устаревшая общая версия останавливает дельту
  без частично установленного локального обновления.
- Включён ремонт merge-шага из 5.23: все одноимённые результаты параллельных партий объединяются
  в уникальный dependency bundle вместо чтения только первой партии.

## 5.23 — 2026-07-22

- Выходы параллельных/декомпозированных шагов теперь материализуются для merge-шага как один
  `upc-dependency-bundle/1.1`: уникальное имя содержит id исходного шага, а manifest сохраняет
  точный `step_id`, kind, исходный путь, локальный путь, размер и SHA-256 каждого артефакта.
- Исправлен дефект, при котором generated-эксперт получал родительскую папку только первого из
  нескольких одинаково названных `step_result.json` и мог молча объединить одну партию вместо всех.
  Принятые независимые партии при точечном ремонте не пересчитываются.
- Добавлена регрессия 4 map-партии × 2 одноимённых артефакта → 8 уникальных файлов и manifest в
  одной входной папке merge-эксперта.

## 5.22 — 2026-07-22

- Установщик больше не смешивает две роли `agent_id`: выбранный пользовательский Qwen остаётся
  reasoning-агентом, а глобальные системные эксперты сохраняются в каноническом scope
  `agent_extella_default`, как их затем читает работающий мост. Это устраняет HTTP 401 на копиях
  Qwen при QA-обновлении.
- Миграция трёх обязательных Universal Process экспертов теперь подтверждается отдельным локальным
  маркером. Если предыдущая дельта успела скопировать новые файлы, но оборвалась до сохранения
  экспертов, следующий запуск не считает обновление завершённым только по номеру версии и безопасно
  повторяет миграцию.

## 5.21 — 2026-07-22

- QA-дельта больше не требует `openpyxl` в системном Python чистого Mac: все XLSX-фикстуры
  универсальной матрицы создаются напрямую как стандартизированный OOXML.
- Мост получил read-only OOXML-профилировщик `.xlsx` без внешних библиотек. Даже если `openpyxl`
  отсутствует, Source Model видит до 12 листов, их имена, колонки и ограниченную выборку строк.
- Добавлен отдельный регрессионный тест, который принудительно блокирует импорт `openpyxl` и
  доказывает работу чистого Mac fallback.

## 5.20 — 2026-07-22

- Строитель теперь работает task-first: Qwen сначала видит цель, фактические входы, ограничения и
  приёмку без технического имени эксперта и выбирает маршрут `reuse`, `build`, `llm_worker`,
  `split_step`, `replan`, `need_human` или `acquire`.
- Решение не требует дополнительного вызова модели: маршрут и Decision Record возвращаются тем же
  reasoning-вызовом Source Model, поэтому новый порядок не увеличивает число LLM-циклов.
- Только после выбора маршрута создаётся технический эксперт. Для `build` Qwen получает требование
  детерминированного кода, для `llm_worker` — runtime Qwen со строгим JSON/evidence-контрактом, а
  `reuse` принимается только с реально найденной capability.
- Выбранный маршрут сохраняется в Task Package, evidence и UPC checkpoint; в интерфейсе шага видно,
  что Qwen решила до создания эксперта и почему.

## 5.19 — 2026-07-22

- Каждый настоящий запуск эксперта сохраняет единый `step_result.json`: status, summary,
  structured data, artifacts, errors и evidence. Харнесс не придумывает обработанные файлы или
  успешные проверки, поэтому приёмка остаётся fail-closed.
- Свободное описание выхода от планировщика (например, «структурированные данные, готовые к
  проверке») больше не считается буквальным именем обязательного файла. Нефайловые результаты
  проверяются через `step_result_json`, а реальные отчёты и документы — отдельно.
- Незавершённые UPC-графы мигрируют на новый контракт без потери уже принятых соседних шагов;
  точечный ремонт продолжает работу с текущего шага.
- Терминальные состояния сборки снимают локальный флаг ремонта и возвращают кнопке нормальный вид:
  после исчерпания бюджета больше не остаётся ложного «Ремонт запускается…».
- QA-дельта дополнительно прогоняет универсальную агентную матрицу и защиту от повторных кликов.

## 1.2.0 — 2026-07-11

Кабинет агента, индивидуальные личности (Qwen) и живые базы знаний. Каждый собранный агент — управляемый объект с обложкой; процессы с правовой основой запрашивают кодекс вживую, а не вшивают статьи в код.

### Живые базы знаний (knowledge grounding)
- **Каталог** (`catalog/catalog.json`, добавлен в репо): новая способность `knowledge_grounding` (реюз `kp_ask`) + секция `knowledge_packs` — Трудовой/Гражданский/Налоговый/Уголовный/Административный/Предпринимательский кодексы РК с доменами.
- **Интервью** (`ui/wizard.html`): новый вопрос `knowledge_base` — на какой закон/регламент опирается процесс.
- **`wz_generate_blueprint`**: правило 2д — если процесс опирается на закон/кодекс, blueprint ОБЯЗАН добавить стадию `knowledge_grounding` и выбрать базу по домену (поле вывода `knowledge_pack.pack_id` + валидация против `knowledge_packs`). Проверено: HR-процесс → стадия «Поиск норм ТК РК» + `trud_rk` (Qwen выбирает сам).
- **Мост** (`ui/server.py`): в `_run_build` — авто-установка нужной базы (`kp_install_pack`) на target прогона + детекция knowledge-стадии; вместо кодогена сохраняется тонкая обёртка `_build_kp_stage` из `_KP_STAGE_TEMPLATE` (реюз `kp_ask`, привязан pack, мягкая деградация без target). Оркестратор (`_ORCH_TEMPLATE`/`_make_orchestrator`) получает `KP_STAGES` и пробрасывает `target`+`api_token` только в knowledge-стадии — `kp_ask` бежит там, где лежит база. Доказано сквозной пересборкой HR-агента: стадии `anonymize → search_labor_code_rk (реюз kp_ask, trud_rk) → generate → deliver`.

### Индивидуальные личности агентов (Qwen)
- **Мост**: `_gen_identity(name, description, experts)` — Qwen по блупринту генерит `emoji/accent/tagline/capabilities/category`. `/x/publish` передаёт их в `wz_publish_pack`.
- **`wz_publish_pack`**: карточка витрины обогащена полями `emoji, accent, category, capabilities, tagline, status, sessionId`; фолбэк-эвристика по ключевым словам, когда Qwen молчит.

### Публикация в витрину — фикс
- **`wz_publish_pack`**: каталог `_mkt_automations` теперь пишется под `agent_extella_default` (владелец общей витрины, под которым её читает тулбар). Раньше запись шла под агентом пака и уходила в его агент-«тень» — карточка в магазине не появлялась. Урок: общие витрины (`_mkt_*`) писать/читать под одним агентом.

### Новый эксперт
- **`wz_agent_runlog`** — серверная история запусков агента (`append`/`list`, KV `agent_runs:<id>`, cap 200). Тулбар (ручной запуск) и планировщик пишут в общий лог.

### Самодостаточный UI Визарда (`ui/wizard.html`, `ui/server.py`)
- Вкладка «Автоматизации» Визарда теперь показывает **hero-обложки** (эмодзи/цвет/слоган из личности) и **полный кабинет агента** (Обзор/Запуски/Расписание/Настройки) — не завися от тулбара. `/x/automations` обогащён личностью (из витрины по `sessionId`), полной историей запусков, `knowledge_pack` и целью. Кабинет переиспользует существующие действия Визарда (запуск/расписание/источник/деплой/шеринг) + показывает «живую базу знаний» и «из чего собран».

### Тулбар (тем же контрактом — см. `HANDOFF_TOOLBAR_v1.2.md`)
- Аналогичный кабинет+обложки есть и в тулбаре (другой репозиторий/владелец). Контракт манифеста карточки — в handoff-доке; Визард и тулбар читают одну витрину `_mkt_automations`.

## 1.1.0 — 2026-07-10

Срез живого состояния после трёх дней работы (07–10.07). Мост поднят с v3.47 до **v3.53**.

### Мост и UI (ui/server.py, ui/wizard.html)
- **Нативная стройка на fine-tune Qwen** (v3.48–3.49): эксперт строится «действием» модели (create, store:true), мост усыновляет его в global и гоняет приёмку на реальном входе. Новые опциональные поля `config.json`: `llm_agent_id` (кодоген), `design_agent_id` (план/чертёж). Без них всё работает на основном `agent_id`.
- **Чат-память per-session** (v3.50): платформа не хранит keyless-разговоры — мост сам ведёт стенограмму `sessions/<sid>_chat.json` и подаёт историю целиком.
- **План/чертёж на Qwen** (v3.51–3.52): design-split, вычищены мёртвые gpt-4o-заглушки. Канон: клиентские агенты — только платформенный Qwen.
- **Фикс вечного спиннера** (v3.53): пропущенный `return` в run_expert-ветке моста.
- **Вкладка Composer + витрина**: новые маршруты `/x/publish`, `/x/automations`, `/x/compose`, `/x/run_flow`, `/x/run_process`, `/x/my_library`, `/x/cap_search`, `/x/cap_install`, `/x/cap_remove`, `/x/configure`; EN/RU-переключатель.

### Эксперты
- Новые: `wz_session_prune` (чистка сессий, опасное действие только при `apply=true`), `wz_publish_pack` («Поделиться» → карточка в витрину), композитор — `wz_auto_compose`, `wz_flow_run`, `wz_capability_search`, `wz_capability_install`, `wz_capability_uninstall` (честное удаление с устройства).
- Обновлены: `wz_build_plan` (max_output_tokens 16000 — фикс обрезки больших планов), `wz_cli_capability_factory` (богатые описания + фразы для семантического поиска), `wz_connector_telegram` (+`mode='send_document'` — отправка файлов), `wz_scheduler_tick` (+обработка входящих через `reply_expert`).
