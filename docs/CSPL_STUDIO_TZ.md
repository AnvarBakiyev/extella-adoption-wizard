# ТЗ: CSPL Studio v0 — «создай свой предметный язык словами» (проект)

**Заказ Анвара (16.07.2026):** «CSPL Studio я очень хочу сделать». Рамка — ТЗ Wizard v2 §10.
Контекст-посыл: enterprise спрашивает «чем вы отличаетесь от Codex?» — Studio и есть ответ:
Codex пишет код в репозиторий разработчика; Extella даёт БИЗНЕСУ собственный исполняемый
словарь на его инфраструктуре — с проверкой, регистрацией, версией и правами. Это категория,
а не фича.

## Ключевой инсайт: половина Studio уже построена
CSPL Extella (fython/$extens + регистрация handlers на листенере) — живой механизм. И у нас
уже есть ДВА детерминированных «компилятора», просто не оформленных как CSPL:
- `_make_orchestrator` (`ui/wz_build.py`): JSON-описание стадий → исполняемый оркестратор.
  Это фактически **pipeline_dsl** из ТЗ §10.3.
- Рендер отчёта в `_ORCH_TEMPLATE` (summary → .md/.xlsx): зачаток **report_dsl**.
Studio v0 = диалоговая обёртка + реестр + sandbox-тест ВОКРУГ этого класса генераторов,
а не стройка компиляторов с нуля.

## Что такое Studio v0 для пользователя
1. **Диалог:** «Хочу описывать отчёты одной страницей: какие колонки, какие итоги, кому» →
   Qwen формирует спецификацию языка: имя, входная схема (JSON Schema), что компилируется
   на выходе (артефакт: xlsx/md/оркестратор/форма), 3 примера-fixture.
2. **Генерация handler:** системный эксперт `cspl_builder_v0` (Qwen-кодоген по жёсткому шаблону,
   как _BUILD_SYS) создаёт: parser+validator (отклоняет некорректную программу ДО исполнения,
   с номером строки) + compiler (детерминированный: одинаковый вход+версия → одинаковый артефакт).
3. **Sandbox-тест:** прогон на fixtures ЛОКАЛЬНО (вертикальный срез, как в стройке); отчёт
   позитив/негатив. Канон вета: чужой/генерённый код не регистрируется без прохождения fixtures.
4. **Регистрация:** handler сохраняется экспертом (global) + запись в `cspl:registry` (KV,
   default-скоуп; версия, схема, fixtures, digest) + строка в Capability Registry (`/x/registry`,
   тип `cspl_handler`) — виден всем четырём поверхностям.
5. **Использование:** программа на новом языке = параметр вызова handler-эксперта; из Чата,
   Визарда (стадия), Композитора (блок), API.

## Этапы
- **S1 — прототип на report_dsl (доказательство конвейера).** Один язык end-to-end: диалог →
  spec → handler → fixtures → регистрация → «программа» владельца (описание отчёта словами/YAML)
  компилируется в живой .xlsx. Реюз готового рендера. Критерий: некорректная программа отклоняется
  до runtime; одинаковый вход даёт байт-в-байт одинаковый отчёт.
- **S2 — pipeline_dsl оформлением.** _make_orchestrator регистрируется как системный handler
  (это признание факта, не новая стройка) → Визард начинает «компилировать процессы» через
  реестровый CSPL — единый словарь для команды и внешних агентов.
- **S3 — Studio-UI.** Иконка на «Рабочем столе» (системная, рядом с Композитором): список
  языков (из cspl:registry), «создать язык» (диалог S1), карточка языка: версии, fixtures,
  где используется. Версионирование: новая версия не ломает старые программы (compatibility
  range, откат).

## Не делаем в v0 (честно)
Grammar-генераторы общего вида, вложенная композиция CSPL-в-CSPL, миграционные хуки, импорт из
Plugin — это §10 целиком, после S1–S3 по спросу. Регистрация handler на «чужих» устройствах —
зависит от мультитаргет-T1 (паспорт: какие handlers уже стоят).

## Риски
- Кодоген слабее на «языках», чем на стадиях → жёсткий шаблон handler + fixtures-гейт (S1 покажет).
- Детерминизм при Qwen-участии → Qwen участвует ТОЛЬКО на этапе создания языка; компиляция
  программ — чистый код без LLM.

**Оценка:** S1 — 1–2 дня; S2 — полдня; S3 — 1 день UI. Старт — по слову Анвара.
