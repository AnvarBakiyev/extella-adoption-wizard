# ТЗ: переосмысление архитектуры Визарда — единый цикл «создать → работать → пересобрать»

Дата: 2026-07-12. Владелец: Анвар (CCO Extella). Автор черновика: рабочая сессия.
Статус: ТЗ на согласование. Сборка — отдельными заходами по разделу «Порядок сборки».

Связанные доки: `REFACTOR_PLAN.md` (гигиена/швы), travel-эталон кабины
`extella-travel-agency-pack/app/onboarding.html` (вкладки Настройка/Продажи/Маркетинг),
пак композиции (`wz_auto_compose` + `wz_flow_run` + `composer:catalog`).

---

## 1. Проблема, которую решаем

Сегодня в Визарде 5 вкладок (Composer / Мои / Мастер / Автоматизации / Процессное ядро),
и они выросли исторически, а не из задачи пользователя. Боли:

- **Автоматизация = «пульт над готовой».** Управлять можно, а **работать и править изнутри — нельзя.**
  Любая мелочь (добавить получателя, поменять текст сводки, сдвинуть порог) гонит пользователя
  обратно через Мастера — это дорого и не по делу.
- **Настройки/подключения** живут отдельным блоком-стеной наверху («подключи всё вперёд»),
  вместо того чтобы быть привязанными к конкретной автоматизации по мере надобности.
- **Создание раздвоено** (Мастер vs Composer) — два входа, делающие по сути одно.
- **Процессное ядро** — легаси CX-демо без пользовательской задачи.

Эталон того, «как надо» — travel-пак: у автоматизации своя **рабочая кабина** с вкладками,
где реально работают (вставить токен, загрузить базу, проверить бота, подобрать туры,
сгенерить договор, отправить), а не только смотрят статус.

## 2. Ключевая идея: один цикл, а не набор экранов

Вся архитектура — это **один жизненный цикл автоматизации:**

```
    СОЗДАТЬ  ─────────►  РАБОТАТЬ / ПРАВИТЬ  ─────────►  ПЕРЕСОБРАТЬ
  (пустая кабина)        (кабина автоматизации)         (обратно в Мастера)
        ▲                                                      │
        └──────────────────────────────────────────────────────┘
```

- **Кабина** — существительное системы. Она определяет, из чего состоит автоматизация.
- **Создание** — это **кабина с пустого листа**, наполняемая двумя дверями (Мастер / Композитор).
- **Композиция** — движок, который по словам заполняет поля кабины (не отдельный раздел).

Отсюда порядок: **сначала кабина** (она фиксирует модель), потом создание (то же, но с нуля),
потом композиция как второй вход в создание.

## 3. Модель автоматизации (существительное)

Автоматизация состоит из полей — это и есть то, что кабина показывает и правит,
а создание заполняет впервые:

| Поле | Что это | Правится в карточке | Технический носитель |
|---|---|---|---|
| Личность | иконка, имя, польза-одной-строкой, статус | да | манифест автоматизации |
| Источник(и) | откуда берутся данные | да (заменить/перепривязать/креды) | `/x/source_bind`, `source_test` |
| Получатели / каналы вывода | кому и куда уходит результат | да (добавить/убрать) | доставки (Telegram/email/…) |
| Параметры | период, пороги, колонки, тон | да | манифест + вход эксперта |
| Шаблон сообщения | текст результата | да (перегенерить через Qwen) | `run_llm_expert` |
| Расписание | когда запускается | да | scheduler / cron |
| Стадии процесса | скелет: шаг 1→2→3 | **нет → Мастер** | собранные эксперты + оркестратор |
| Прогоны | история, логи, доставки | чтение | `/x/monitor` |

**Граница правок (железное правило):**
меняешь **данные / адресатов / текст / параметры / расписание** → **в карточке, без Мастера.**
меняешь **скелет (стадии) или суть процесса** → **через Мастера.**

## 4. Кабина автоматизации — 4 вкладки

По образцу travel-онбординга. Внутри карточки конкретной автоматизации:

1. **Запуск и результат** — крупная «▶ Запустить сейчас», последний результат прямо на странице
   (цифры/сводка + «Открыть»), полоса **Готовности** (что нужно этой автоматизации; зелёное=готово,
   жёлтое=нажми настрой; настройка открывается тут же, не блокирует запуск).
2. **Настройка** — получатели (➕ добавить/убрать), источник (заменить/креды), параметры,
   расписание. Всё редактируемое inline. Кнопка «Переделать через Мастера» — на месте, нужна редко.
3. **Шаблон сообщения** — правка/перегенерация текста результата через Qwen
   («сделай короче», «добавь сравнение с прошлой неделей») с превью.
4. **Прогоны** — история запусков, логи, что и кому ушло.

**Настройки — баланс:** не спрятаны и не вынесены вперёд. Живут на самой автоматизации как
«Готовность» + вкладка «Настройка»; показывается только то, что нужно **этой** автоматизации,
по мере надобности. Общий список всех подключений — спокойная отдельная вкладка «Настройки»
уровня приложения (заходят редко), а не стена наверху.

## 5. Список «Мои автоматизации» (уровень выше кабины)

Каждая автоматизация в списке = мини-карточка: иконка + имя + польза + статус готовности (2/3)
+ быстрый «▶». Клик → кабина. Прежний блок «🔌 Подключения» с верха **убирается**
(подключения переезжают внутрь готовности каждой автоматизации + вкладку «Настройки»).

## 6. Создание = один вход + умная маршрутизация (Композитор-первым)

Слияние Мастер + Composer. Создание — не отдельный продукт и **не две двери на выбор**
(пользователь не знает заранее, нужен ли новый код или хватит проводки). Это **один вход
«Создать» + один мозг, который маршрутизирует:**

```
Задача словами → Композитор пробует собрать из библиотеки (быстро, без кодогена)
   ├─ хватает блоков?  ДА → сборка flow → кабина
   └─ есть пробел?     НЕТ → Мастер: интервью → КОДОГЕН недостающего эксперта
                              → кладёт его в библиотеку → та же сборка → кабина
```

- **Композитор** = **быстрый режим** единого Создания: «расставь то, что уже в библиотеке».
  Одна фраза → подбор блоков + модель → заполнил поля кабины (§3) → подтверждение. Без кодогена.
  Это `wz_auto_compose` как режим, а не отдельная вкладка. **Повседневный путь.**
- **Мастер** = **глубокий режим**: включается на ПРОБЕЛ — когда в библиотеке нет нужной способности.
  Интервью → кодоген недостающего эксперта → **эксперт попадает в библиотеку** → дальше Композитор
  доводит сборку. Дорогой запас для настоящей новизны.

**Явный вход в Мастер остаётся** (кнопка «пройти интервью») для сложного процесса, когда человек
сам хочет глубоко. Но по умолчанию система решает сама. Обе ветки приземляют в кабину §4.

Соответствует тезису **Install → Adapt → Compose → Deploy**: композиция — повседневность,
кодоген — дорогой запас.

## 7. Библиотека — общее знание обоих режимов

Библиотека = что **установлено** + что **можно установить** (наши плагины, эксперты, скиллы,
MCP-репо через `wz_programs_harvest`). Оба режима обязаны её знать, по-разному:

- **Композитор** библиотекой живёт — вся его работа «что уже есть / что поставить»; без неё он пустой.
- **Мастер** тоже знает библиотеку — чтобы **переиспользовать** (он это уже умеет через
  `wz_build_plan` — реюз экспертов), а кодогенить **только настоящий пробел**, не строя заново готовое.

Формула: **Композитор — «расставь то, что в библиотеке»; Мастер — «дострой библиотеку, потом расставь».**
Композиция не имеет своего экрана — она наполняет модель §3.

## 7bis. Доменный кабинет — у КАЖДОЙ автоматизации (кадр Анвара 12.07, вечер)

Инсайт с Travel-пака: настоящая ценность кабинета — не управление (запуск/расписание), а
**доменная работа**: лиды, подборки, черновики, правила, доп. поля. Целевая модель:

- **Кабинет = каркас + доменная вкладка.** Универсальный каркас (Запуск · Настройка · Шаблон · Прогоны)
  один для всех; у автоматизации с доменной панелью она встраивается ПЕРВОЙ вкладкой «🧳 Работа»
  (iframe родной панели пака; НЕ отдельное окно — матрёшка «кабинет в кабинете» запрещена).
  Сделано для Travel (panel_url/panel_name в сессии).
- **Лестница к «кабинету у каждого»:**
  1. ✅ Ручные панели флагманских паков встраиваются вкладкой (Travel).
  2. Generic-вкладка «Правила и поля»: бизнес-правила словами + доп. поля per-automation,
     хранятся в сессии, подмешиваются в шаблон/промпт процесса. Мелкие правки — тут.
  3. **Генерируемая доменная панель**: Строитель при стройке генерит panel-манифест из blueprint
     (формы полей, действия, списки) → у каждой автоматизации свой «Travel-уровень» кабинет.
- **Граница остаётся**: мелочь (поля/правила/текст) — в кабинете; сильная переделка —
  «возвращаясь к Визарду» (Мастер).

## 8. Что убираем

- **Процессное ядро** — вырезаем (легаси CX-демо, нет пользовательской задачи).
- **Composer как отдельная вкладка** — сливается в «Создать» (быстрый режим единой маршрутизации).
- **Блок «Подключения» наверху Автоматизаций** — переезжает в готовность/Настройки.

Итоговая навигация (цель): **Создать · Мои автоматизации · Настройки.**
(«Мои» библиотека возможностей — пока оставляем в Визарде; позже мигрирует в тулбар.)

## 9. Порядок сборки (отдельными заходами)

1. **Кабина автоматизации** — сперва вкладка «Настройка» (правки: получатели/источник/параметры/
   расписание без Мастера — самая частая боль), затем «Запуск и результат», «Шаблон», «Прогоны».
   Это фиксирует модель §3 на живых автоматизациях.
2. **Список «Мои автоматизации»** в новом виде + перенос подключений в готовность/Настройки.
3. **Создание = один вход**, слияние Мастер+Composer; сперва глубокий режим (Мастер: интервью→кодоген→кабина).
4. **Быстрый режим (Композитор)** — `wz_auto_compose` + маршрутизация «реюз-первым, Мастер на пробел»;
   библиотеко-осознанность обоих режимов.
5. **Чистка**: удалить Процессное ядро, старый Composer, старый блок Подключения.

Технически кабина не требует нового движка: получатели/источник/расписание — те же привязки,
что Визард уже пишет в сессию; правки из карточки дёргают существующие эндпоинты
(`/x/source_bind`, schedule, доставки), перегенерация шаблона — `run_llm_expert` на Qwen.
Кабина = **редактируемый доступ** к тому, что уже под капотом.

## 10. Инварианты (канон, не нарушать)

- Весь клиентский LLM — **только Qwen** (перегенерация шаблонов, композиция), никогда Claude.
- Правки из карточки пишут решение в сессию (`~/extella_wizard/sessions/`), как и Мастер.
- Продовые агенты заморожены (F2); письма/внешняя запись — черновики, отправляет человек.
- Скоупинг per-agent; общее — `global: true`.
