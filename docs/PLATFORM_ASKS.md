# Вопросы-блокеры к платформе api.extella.ai (для Тимура)

Составлено 17.07.2026 по фактам стройки визарда/Кабинета/Studia (репо: v3.99). Каждый пункт —
воспроизводим; где есть — указан наш обходной путь (workaround), который хотим убрать.
Контекст: ТЗ Wizard v2 (`EXTELLA_WIZARD_V2_FINAL_TZ.md`) + маппинг `docs/WIZARD_V2_MAPPING.md`.

## А. Стратегические (блокируют P0 из ТЗ v2)

### A1. Версионирование экспертов / immutable Release
`POST /api/expert/save` молча перезаписывает код; версий и истории нет.
**Блокирует:** release-lockfile (ТЗ §8.9.10), безопасный реюз блоков между клиентами,
платформенный откат (наш — самодельный, через снапшоты blueprint в сессии).
**Вопрос:** планируется ли версионирование (save → новая версия; run с pin версии; история)?

### A2. Account-shared скоуп KV / платформенный Capability Registry
KV скоупится по `X-Agent-Id`: один ключ под разными агентами = разные записи («тени»).
Стреляло минимум трижды (витрина `_mkt_automations`, каталоги, тесты). Наш Capability Registry v0
живёт на КОНВЕНЦИИ «всё общее — под agent_extella_default» — это дисциплина, не гарантия.
**Вопрос:** возможен ли настоящий shared-скоуп KV уровня аккаунта (без агент-привязки) —
или платформенный registry-эндпоинт для общих каталогов?

### A3. Паспорт и heartbeat листенера (мультитаргет)
Листенер о себе не публикует ничего; паспорта/пульс мы сделали поверх (wz_target_passport +
пере-паспорт тиком раз в 6ч). Вопросы из `docs/MULTITARGET_TZ.md`:
1) может ли листенер публиковать паспорт/heartbeat сам (при старте + периодически)?
2) стабилен ли `target_id` между переустановками листенера?
3) есть ли API списка targets аккаунта с их онлайн-статусом?
**Блокирует:** честный failover (перенос стадии на живое устройство), авто-карту устройств.

### A4. Коллаборация между аккаунтами (организации / шаринг / роли)
Запрос Анвара: «работать с другими людьми, у кого есть Extella, вместе». Сегодня между
аккаунтами возможен только обмен ПАКАМИ (копия процесса разворачивается у коллеги install.py'ем);
совместной работы над ОДНИМ живым процессом нет — ни общих витрин, ни ролей (Business Owner /
Operator / Reviewer из ТЗ §6), ни исполнения стадии на устройстве другого аккаунта.
**Вопросы:** 1) есть ли/планируются организации (несколько пользователей в одном контуре с ролями)?
2) что такое X-Profile-Id и профили (create_profile существует) — можно ли использовать профили
как «людей» внутри аккаунта с разделением прав? 3) возможен ли шаринг ресурса (процесс/эксперт/KV)
между аккаунтами по приглашению?
