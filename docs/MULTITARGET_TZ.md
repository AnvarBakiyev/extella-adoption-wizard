# ТЗ: Мультитаргет — «процесс живёт на нескольких устройствах» (v1, проект)

**Заказ Анвара (16.07.2026):** «надо продумать, как работаем с другими устройствами — фичу хочется
реализовать». Рамка — ТЗ Wizard v2 §13 (Target-aware execution), но по нашему правилу малой крови:
сначала видимость и честность, потом планировщик.

## Что уже есть (факты, не планы)
- Targets платформы: регистрация устройств, `target` в expert/run пинит исполнение; VPS-хостинг
  (`HOST_TARGET`) гоняет расписания 24/7; kp-стадии уже пинятся на устройство базы знаний.
- Проверенные классы устройств: Mac владельца (мост+листенер), VPS PS.kz (прод-хостинг),
  Windows VM (1С через COM в subprocess), второй аккаунт.
- Известные грабли (= зачем паспорт): brew скрыт от PATH листенера; comcntr виснет в листенере;
  CSPL-handlers надо регистрировать на каждом новом устройстве; листенер headless (без GUI/pyobjc).

## Целевая картина v1 (три этапа)

### T1 — Паспорт устройства + карта (малая кровь, видимость)
- Эксперт `wz_target_passport`: запускается НА устройстве (через target), собирает честный паспорт:
  ОС/арх, python/runtime, доступные приложения (Excel/1С/офис — по известным путям), локальные модели
  (ollama tags), зарегистрированные CSPL-handlers, свободное место, время. Пишет в KV
  `target:passport:<target_id>` (global, default-скоуп) + `passport_at`.
- Кабинет процесса: блок «Где исполняется» — карта «стадия → устройство» из фактов сборки
  (оркестратор/kp-стадии/расписание) + свежесть паспорта. Реестр возможностей (`/x/registry`)
  дополняется типом `target`.

### T2 — Требования стадий + preflight
- Blueprint: стадия получает `target_requirements` (нужна 1С / локальная папка / local_only-данные).
- Мост перед прогоном/расписанием: preflight — паспорт свежий? требуемое приложение на месте?
  handlers зарегистрированы? Нет → честный отказ с remediation («запустите онбординг устройства»),
  как sample_preflight (WZ-07-паттерн).
- `local_only`-данные: стадия не покидает устройство (сегодня это канон вручную — станет проверкой).

### T3 — Failover-политики v0
- Пер-стадийно: `queue_until_online` (дефолт — ждать своё устройство), `equivalent` (только чистые
  стадии без side effects), `hard_bound` (1С, локальные папки). Решения фиксируются в run-record.

## Не делаем в v1 (честно)
Скоринг/оптимизация размещения, attestation/подпись паспорта, drag-and-drop карта, heartbeat в
секундах — это платформенный уровень (листенер), см. вопросы ниже.
