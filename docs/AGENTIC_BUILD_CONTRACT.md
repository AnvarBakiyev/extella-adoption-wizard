# Агентная стройка сложных процессов (as-built, v4.96)

## Когда включается

`wz_build._run_build` переводит задачу в `build_mode=agentic`, если у неё несколько одновременно
используемых файлов либо нелинейный DAG. Линейная одновходовая стройка сохраняет прежний путь.

## Контракт

1. `wz_agentic.make_task_package` собирает без потери структуры ответы интервью, комментарии,
   решения, правила/поля, blueprint, build plan, все профили файлов и доступный capability-каталог.
2. Qwen создаёт один исполняемый CSPL/fython-эксперт с фиксированной сигнатурой.
3. Код не допускается к запуску, если содержит значения контрольного образца, абсолютные пути,
   shell или fallback-поиск вне `source_file`.
4. Визард, а не строитель Qwen, запускает эксперта с явными `source_file` и `output_dir`.
5. Детерминированная приёмка требует доказательства обработки каждого входного файла,
   содержательную summary, acceptance checks и открываемые `report.md`/`report.xlsx`.
6. Независимый Qwen-судья проверяет результат против Task Package. При ошибке фактический результат
   возвращается строителю; максимум три попытки solve → run → repair.
7. Только принятая версия записывается в build с `agentic_contract=1` и может запускаться из кабинета.

Доказательства лежат в `runs/<build_id>/task_package.json` и `agentic_evidence.json`.

## Исполнение и упаковка

Каждый ручной прогон получает отдельную папку `runs/process_runs/<session>/<run_id>`; затем отчёты
переносятся штатным persistence-слоем. `/x/deploy_agent` синхронизирует в выбранного агента один пакет:
эксперты сборки, tagged concepts процесса и tagged rules владельца. Повторный deploy применяет diff.

Создание новой «копии Qwen» через `/api/agent/create` обязательно проходит реальный smoke-test.
На текущем API новая Alibaba-копия может оказаться Pro custom-agent и потребовать BYOK
(`pro_key_required`); такую пустую карточку визард удаляет и предлагает привязать уже работающего
Qwen. Это ограничение платформенного API, а не повод изображать готового агента.

## Честное ограничение v4.96

Локальный multi-file bundle работает вручную на устройстве владельца. Расписание 24/7 для такого
каталога блокируется кодом `local_bundle_schedule_unsupported`, пока не реализован device-local
планировщик либо материализация всего bundle на VPS.
