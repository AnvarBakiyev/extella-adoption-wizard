## 1. Диагноз (проверено по коду, не ощущения)

| Боль | Факт в коде |
|---|---|
| «Не знаю, как отключить CI» | Кнопки Пауза/Выключить **нет вообще**. Статус в шапке — бейдж из витринной карточки (`_wzStatus` ← `identity`), к реальности не привязан. Остановить можно только сняв расписание (теряются настройки) или удалив. |
| «Travel хорош, CI — нет» | Travel — ручная доменная панель («🧳 Работа»), там **делают**: вставляют токен, подбирают тур, шлют черновик. Генерируемый `🧭 Процесс` CI — статичные карточки этапов: витрина для чтения, а не рабочее место. |
| «Композиция — будущий продукт, а кабинета нет» | После «Сохранить» composed-flow падает в общий кабинет и **теряет чат-доводку** (чат живёт только на черновике). Машина публикации (`wz_publish_pack`) есть, но **композиции публиковать не умеет** (см. §5.2). |
| Скрытый живой баг (нашло ревью) | «Удалить» архивирует сессию и снимает расписание, но **НЕ снимает входящие**: `inbound:<sid>`, индекс и `hookmap:` остаются — удалённая автоматизация продолжает принимать и отвечать в Telegram. Чинится в C1. |

**Удешевляющие факты:** тик уже уважает `active` в расписаниях и входящих; деплойнутая копия тика
= репо (сверено). **Удорожающие факты (ревью):** дайджесты прогонов по расписанию НЕ сохраняются
(пишутся в локальный файл исполняющего устройства — для расписания это VPS); `installed`/`missing`
композиции живут только в эфемерном ответе; каждая пересборка flow рождает новый `flow_id` + карточку.

## 2. Рамка: кабинет = три слоя

| Слой | Вопрос человека | Сегодня |
|---|---|---|
| **Работа** | «Что мне сделать в моей работе?» | Travel ✅ · остальные ❌ |
| **Пульт** | «Работает ли оно, как выключить, когда бежит, всё ли здорово?» | ни у кого целиком |
| **Эволюция** | «Как доработать и превратить в продукт?» | только черновик Композитора ⚠️ |

Целевое: все три слоя у любой автоматизации; наполнение — по архетипу и происхождению
(Мастер / Композитор / пак), скелет один.

## 3. Слой «Пульт»: статус-модель, Пауза, здоровье

### 3.1. Статус-модель (источник истины — реальность, не витрина)

| Статус | Определение (вычисляется) | Что видит человек |
|---|---|---|
| `active` | есть расписание `active:true` ИЛИ включённые входящие | ● работает · «след. запуск через N ч» |
| **`attention`** | последний прогон упал / креды истекают ≤7 дн. или истекли / доставка не дошла | 🟡 «нужна помощь» + **человеческая причина + ОДНА кнопка-действие** («Обновить токен» → открывает нужное поле; «Прогон упал» → причина словами + «Запустить снова») |
| `paused` | расписание/входящие есть, но `active:false` | ⏸ на паузе · «Возобновить» |
| `manual` | ни расписания, ни входящих | ○ вручную · «Поставить на расписание» |
| `running` | прогон идёт прямо сейчас | ⏳ «прогон идёт (N мин)»; кнопка запуска заблокирована; по завершении тост «готово, открыть результат» |

- `attention` — **обязателен в C1**: главный реальный вопрос владельца — не «как выключить», а «CI три
  дня молчит — он сломан или так и надо?». Данные уже есть в `/x/monitor` (желтит токены заранее) —
  завести их в шапку кабинета и в карточку грида.
- `frozen` (канон F2) — не состояние жизни, а **пометка для режима разработчика**; клиенту не показываем
  (решение по умолчанию, вопрос №4 остаётся).
- Витринная карточка (`identity.status`) — **best-effort кэш** (истина в sched/inbound KV; потеря
  записи допустима, пауза от успеха этой записи не зависит — там RMW общей ленты `_mkt_automations`).

### 3.2. Пауза / Возобновить (стройка C1)

- Кнопка в шапке: `⏸ Пауза` / `▶ Возобновить`. Эндпоинт `POST /x/pause {session_id, paused}` *(предложение)*:
  `sched:<sid>.active`, `inbound.active`, **и `hookmap:<route_token>.active`** (webhook-режим живёт в
  отдельном ключе — гасить оба, иначе события копятся мимо паузы).
- **Ничего не удаляется**; Resume возвращает как было. Семантика Resume: `next_due_ts := now + interval`
  (без catch-up-шторма); на паузе строку «след. запуск» прячем.
- **Входящие на паузе — явная семантика (ревью):** при нажатии Паузы у процесса с входящими —
  предупреждение человеческим языком: «входящие обращения не будут обрабатываться». События копятся
  (`inbq`), но на Resume **бэклог НЕ отвечается автоматически**: offset скручивается вперёд, очередь
  дренируется без ответов, человеку показывается «за время паузы пришло N обращений — обработать?».
  Для Travel-класса это потеря лидов — предупреждение обязательно.
- Ручной «▶ Запустить сейчас» доступен и на паузе (с пометкой «разовый прогон») — вопрос №1 остаётся.
- **Удалить (фикс живого бага):** `automation_delete` дополнительно снимает `inbound:<sid>`, индекс
  `inbound:__index__` и `hookmap:` — как ветка `mode=off` в `/x/inbound`. Обязательная часть C1.

### 3.3. Пульт-приборка (в C1)

- «След. запуск: …» из `next_due_ts` (кроме паузы).
- **Полоса Готовности** (источник / получатели / расписание — зелёное/жёлтое): уже описана в ARCH_TZ,
  возвращается в шапку Пульта (ревью: была потеряна).
- Одна каноническая кнопка «▶ Запустить сейчас» — **только в шапке**. Виджеты «Работы» могут дублировать
  её лишь как алиас той же операции (не три разные кнопки).

## 4. Слой «Работа»: доменный вид по архетипу

Принцип: **кабинет собирается из виджетов, как процесс — из экспертов**. Архетип blueprint определяет
набор; `panel_manifest` расширяется до манифеста виджетов. Ручная панель пака (`panel_url`) всегда
побеждает генерируемую (Travel не трогаем).

**Требование ревью — виджеты с ДЕЙСТВИЯМИ, не витрина для чтения.** Манифест виджета обязан содержать
не только «что показать», но и «что можно сделать» — минимум 2–3 доменных действия на архетип.
Критерий приёмки C3: *владелец совершает доменное действие, не покидая вкладку «Работа»*.

| Архетип | Показ | Доменные действия |
|---|---|---|
| research_digest (CI) | последний дайджест · источники и статус · knowledge gaps | **добавить/убрать конкурента-источник** (пишет в конфиг пайплайна) · замьютить шумный источник · «раскрыть тему глубже» (gap → в следующий прогон) |
| reconcile (дебиторка) | сводка расхождений · динамика прогонов | отметить расхождение «в работе/решено» · порог существенности |
| evaluation (кадровые) | последний отчёт · критерии | загрузить документ на оценку · вкл/выкл критерий |
| inbound-конвейер | родная панель пака; иначе последние входящие + статусы | ответить черновиком · сменить статус лида |
| fallback | текущий `🧭 Процесс` + поля `panel_manifest` | — |

**Нулевое состояние (обязательный раздел — ревью):** свежесобранная автоматизация (runs=[]) показывает
не пустые виджеты, а: (1) «что эта автоматизация будет делать» — одной фразой из blueprint;
(2) полосу Готовности; (3) одну большую кнопку «Сделать первый прогон». Это закрывает и старый
демо-долг «сырой экран после Собрать».

**Честная стоимость (ревью):** «последний дайджест» по расписанию сегодня НЕ сохраняется (уходит в
локальный файл VPS; в run-записи только счётчики). C3 включает **стройку персистенции**: оркестраторы/тик
пишут усечённый `digest_md` в KV (`digest:<sid>` или в run-запись, с лимитом размера; хранить последние
N) + перепрошивка экспертов. Это не «только UI-слой».

## 5. Слой «Эволюция»: чат и кабинет композиции

### 5.1. Чат автоматизации (C4) — единое окно допиливания

Форма: **док-бар снизу на всех вкладках** (рекомендация; вопрос №2 остаётся). Маршрутизация по глубине:

| Глубина | Пример | Куда | Механика |
|---|---|---|---|
| Лёгкая | «шли ещё Гульжан» | recipients/schedule/template (парсер намерения → подтверждение → запись) | эндпоинты есть; парсер — построить |
| Средняя | «добавь поле приоритет и учитывай» | правка полей + пересборка (= W1) | построить |
| Сильная | «добавь этап согласования» | Строитель + запись в сессию (F2) | Строитель есть; handoff — построить |
| Композиция | «убери шаг OCR» | `compose_chat` по flow | есть (черновик) → C2 |

**Канон пересборки (ревью, три железных правила):**
1. На время пересборки автоматизация автоматически `paused` (иначе VPS-тик запустит полусобранное); после — прежний статус.
2. Пересборка не затирает, а **добавляет запись в `builds[]`** → в кабинете кнопка «Откатить к предыдущей версии» (история уже хранится — нужна кнопка).
3. После пересборки — **пробный прогон с показом «до/после»**, и только потом расписание продолжает бежать.

**Переходная честность до W1 (ревью, канон «без фейка»):** пока средняя правка не построена, поле после
сохранения помечается: *«сохранено, но пока не влияет на прогоны — чтобы применить, нужна пересборка
(кнопка → Строитель)»*. UI, изображающий применение правки, которой не было, — запрещён.

### 5.2. Кабинет композиции (C2)

Вкладки поверх общего скелета:

| Вкладка | Что | Честная механика (после ревью) |
|---|---|---|
| **Состав** | из чего собрано · что доустановлено · чего не хватает | `steps` есть в `flow:<id>`; **`installed`/`missing` сегодня НЕ персистятся** (живут в эфемерном ответе) → C2 включает запись их в `flow:<id>` при компоновке (правка `wz_auto_compose` + перепрошивка) |
| **Доводка** | чат-верстак по сохранённой композиции | **Решение: стабильный `flow_id`** — научить `wz_auto_compose` перезаписывать переданный `flow_id` (новый параметр) вместо генерации нового на каждое сообщение. Это убирает: мусор карточек в витрине (сегодня карточка на каждое сообщение чата!), каскад обновлений sched/сессии, и гонку с тиком. Плюс: расширить merge-whitelist тика (`flow_id`, `agent_id`, поля Пульта) + перепрошивка |
| **Продукт** | «Опубликовать как пак» | **Блокер ревью: `wz_publish_pack` композиции НЕ умеет** — пак собирается только из платформенных экспертов по именам; для flow-сессии в `experts` лежат человеческие ярлыки чипов → пак-пустышка. Отдельная стройка: сериализация `flow:<id>` в пак + install.py, воссоздающий flow и ставящий блоки; запись block-id (не ярлыков) при `/x/flow_save`. **До этого — гейтить кнопку publish для composed-сессий** (сегодня она доступна и сломана) |

**Видимость «Продукта» (изменено после ревью):** по умолчанию — только автоматизациям происхождения
Композитор и в режиме креатора; для клиентских Мастер-процессов скрыта (иначе мёртвая кнопка ломает
Travel-ощущение). Мысль «это можно продать» сеем строкой в Составе: «эта сборка может стать паком».

## 8. Канон-инварианты

- Клиентская LLM — только платформенный Qwen (`qwen_agent()`).
- Продовые агенты заморожены (F2): сильные правки — только Строитель с записью в сессию.
- Внешняя запись/письма — только черновики; отправляет человек.
- Пауза не удаляет; Удалить = архив **и снимает входящие** (C1).
- **Без фейка:** UI не изображает применение правки, которой не было (переходные пометки до W1).
- Общий KV — `global:true`; сессии/sched/flow — per-account; identity.status — best-effort кэш.
- Публикация наружу — только по прямому слову Анвара.
