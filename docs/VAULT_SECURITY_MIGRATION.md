# План: закалка хранилища секретов (миграционно-опасная часть)

Статус: **ждёт решения Анвара.** Это то, что НЕ входит в безопасный батч (коммит 86e6d2a),
потому что меняет то, как шифруются/доставляются **живые секреты на боевом VPS**. Ошибка здесь
= осиротить все существующие подключения (клиент вводил токены заново). Поэтому — только осознанно.

Источник: сквозной аудит коннекторов, 4 находки из 15 (#1 critical, #2/#5 high, #14 medium).

---

## Что не так сейчас (простым языком)

Секреты клиента (токены Telegram/CRM, пароли БД) лежат в KV **зашифрованными**. Ключ шифрования —
файл `vault.key` на устройстве. Проблема в том, **как этот ключ рождается**:

1. **#1 (критично) — ключ выводится из PIN, а «соль» публичная.**
   `vault.key = PBKDF2(PIN, соль = sha256("extella-vault:" + client))`. PIN короткий (мин. 6 символов),
   а соль вычисляется из имени клиента (не секрет). Значит, кто получил шифртекст из KV, может
   **перебрать PIN офлайн** (пространство 6 цифр = миллион вариантов — часы на ноутбуке) и расшифровать
   все секреты. PBKDF2 с 600k итераций замедляет, но не спасает при таком коротком PIN.

2. **#2 (высокая) — PIN уезжает на платформу.**
   Провижининг ключа (`wz_vault_provision`) принимает `pin` как **параметр эксперта** и гоняет его через
   `/api/expert/run` → платформа пишет параметры в call-записи. PIN, из которого выводится ключ, попадает
   в логи вызовов.

3. **#5 (высокая) — слабый минимум PIN, нет rate-limit, наружу отдаётся кусок хеша ключа.**
   6 символов мало; провижн можно дёргать без ограничений; ответ содержит `key_sha256[:16]` (утечка для
   сверки перебором).

4. **#14 (средняя) — токен аккаунта уезжает на платформу.**
   Мост шлёт `api_token = CONFIG["auth_token"]` в параметрах коннектора (`/x/connector_test|send`,
   `/x/source_*`) → платформа пишет его в call-записи. Токен аккаунта утекает в экспорт вызовов.

---

## Почему не сделал сразу

Всё это трогает **живой vault.key на боевом VPS PS.kz** и **живой токен аккаунта**:
- Сменить схему ключа = **перешифровать все существующие секреты** (иначе старый шифртекст не
  расшифруется — подключения умрут). Это разовая миграция «расшифровать старым → зашифровать новым».
- Убрать токен из params = коннектор должен брать токен **локально из config.json на хосте** + после
  этого **ротировать старый токен** (иначе уже утёкшие в логи остаются валидными). Ротация ломает всё,
  что использует старый токен, — это только твоё решение.

Канон проекта прямо это защищает: «Продовые агенты заморожены», «секреты в логи не печатать»,
«не переустанавливать VPS без запроса».

---

## Три варианта решения (выбери)

### Вариант A — «Случайный ключ, раздаём файлом» (рекомендую)
- Отказаться от вывода ключа из PIN. `vault.key = Fernet.generate_key()` (128-бит случайный).
- Ключ раздаётся **out-of-band**: копируем файл по SSH на хост (`~/.ssh/extella_hosting_ed25519` уже есть),
  клиенту — через защищённый канал, не через `/api/expert/run`.
- PIN больше не нужен для крипты (можно оставить как локальный «замок» UI, но не как ключ).
- **Плюс:** перебор невозможен в принципе. **Минус:** нужно надёжно доставлять/бэкапить файл-ключ
  (потерял ключ — потерял секреты; но это честная модель).

### Вариант B — «Сильная парольная фраза + случайная соль»
- Оставляем PIN-подобную модель, но: минимум — **парольная фраза** (не 6 цифр), соль — **случайная
  per-install**, лежит файлом рядом (не выводится из client). PIN вводится **локально на хосте**, не
  через платформу.
- **Плюс:** не надо таскать файл-ключ. **Минус:** безопасность упирается в дисциплину пароля пользователя.

### Вариант C — «Device-bound ключ»
- Ключ привязан к устройству (Keychain/DPAPI/файл с жёсткими правами), секреты не обязаны совпадать
  между устройством клиента и хостингом. **Плюс:** сильнее всего. **Минус:** секрет не переносится между
  устройствами без явного экспорта — меняет UX мультитаргета.

---

## План миграции (общий для A/B, когда выберешь)

1. **Заморозить запись секретов** на время миграции (короткое окно).
2. Скрипт на хосте: для каждого `sec:<client>:<connector>` — `plain = Fernet(old_key).decrypt(ct)` →
   `ct2 = Fernet(new_key).encrypt(plain)` → записать во временный ключ `sec2:...`.
3. Проверить, что все секреты перешифрованы (счётчик == индекс), только тогда **атомарно** заменить
   `sec:*` на `sec2:*` и `vault.key` на новый.
4. Прогнать `connector_test`/`source_test` по каждому подключению (read-only) — все `ok:true`.
5. Откат: старый `vault.key` держим в бэкапе до подтверждения зелёного прогона.

## План по токену (#14) — отдельный, тоже с твоим «да»
1. Добавить коннекторам локальный фолбэк: если `api_token` пуст/placeholder — читать из
   `config.json` на хосте (канон: секреты фолбэком из config.json). *(Код-часть безопасна, аддитивна.)*
2. Мост перестаёт слать токен в params (шлёт пусто).
3. Прогнать коннектор на VPS (read-only validate) — должен работать без токена в params.
4. **Ротировать** старый токен аккаунта (твоё действие — ломает всё на старом токене; сначала обнови
   config.json на всех хостах новым токеном, потом отзови старый).
