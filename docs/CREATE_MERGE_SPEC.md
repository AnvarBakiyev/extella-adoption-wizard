# Спец: «Создание» — слияние Мастер + Composer в один вход

Дата: 2026-07-12. Родительский документ: `WIZARD_ARCH_TZ.md` §6–7 (один вход, реюз-первым).
Статус: спец на согласование. Сборка — отдельными инкрементами по разделу «Порядок».

## 1. Что есть сейчас (факты из кода)

Две точки входа порождают **разные артефакты** — это главный узел слияния:

| | Вход | Эндпоинт | Эксперт | Артефакт | Запуск |
|---|---|---|---|---|---|
| **Composer** | одна фраза | `/x/compose` | `wz_auto_compose` | **flow** (`flow_id`) | `/x/run_flow` → `wz_flow_run` |
| **Мастер** | интервью (8 ответов) | `/x/session`,`/x/blueprint`,`/x/build` | Строитель + `wz_build_*` | **процесс** (сессия + оркестратор) | `/x/run_process` |

Ключевые находки:
- **Сигнал пробела уже есть.** `wz_auto_compose` возвращает `missing` — блоки, которых нет в библиотеке
  клиента. Composer уже показывает их («не в твоей библиотеке: …») и предлагает «🌐 Найти в интернете»
  (`/x/cap_search`), но **НЕ передаёт в Мастера достроить** недостающее.
- **Два вида автоматизаций.** Composed-flow (`flow_id`, стор `COMPOSED`) и built-процесс (сессия+оркестратор).
  Кабина §4 (Запуск/Настройка/Шаблон/Прогоны) сейчас построена вокруг **процессов** (session_id).
  Composed-flow в неё полноценно не ложится (нет session_id, другой запуск/доставка).

## 2. Целевая модель (из ТЗ + кадр Анвара 12.07: «задача vs процесс»)

**Ключевое различие дверей — не «быстро vs глубоко», а ТИП РЕЗУЛЬТАТА (намерение пользователя):**
- **Задача → Композитор.** «Найди подписки», «сожми PDF», «дайджест конкурентов»: собрал из готовых
  блоков → запустил → результат. Лёгкая вещь, живёт как flow.
- **Бизнес-процесс → Мастер.** «Дебиторка 28 филиалов каждый понедельник директору»: интервью — это
  выяснение устройства бизнеса (источники, ответственные, расписание, критерий результата). Итог —
  процесс с обвязкой + продовый агент 24/7.

**Мастер — НЕ fallback Композитора.** Это две РАВНОПРАВНЫЕ двери одного входа. Вход при этом один:
(1) пользователь не знает нашей таксономии — он думает «хочу, чтобы делалось само»;
(2) граница размыта («еженедельная сводка» может быть и тем и другим);
(3) обе двери приземляются в одну кабину.

Один вход **«Создать»** + маршрутизация «реюз-первым»:

```
Задача словами → Композитор (wz_auto_compose): собрать из библиотеки
   ├─ missing пуст   → flow готов → «▶ Запустить» / «Сохранить как автоматизацию» → кабина
   └─ missing непуст → развилка:
        • «🌐 Найти в интернете» (есть) — доустановить готовый блок → пере-компоновка
        • «🧩 Достроить Мастером» (НОВОЕ) — интервью ТОЛЬКО по пробелу → кодоген эксперта →
          эксперт в библиотеку → пере-компоновка тем же Композитором → кабина
Явная кнопка «пройти полное интервью» (Мастер с нуля) — для заведомо сложного/нового процесса.
```

## 3. Узел артефакта (решить до сборки)

Composed-flow и built-процесс должны сойтись в **одну кабину**. Три варианта:

- **A. Композитор пишет процесс-сессию** (рекомендую позже): результат `wz_auto_compose` заворачивается в
  сессию с «оркестратором = flow» → сразу ложится в кабину §4 (Запуск/Настройка/Шаблон/Прогоны),
  доставка/расписание/получатели работают как у built. Больше работы в мосту, но единый объект.
- **B. Кабина учит второй тип** (flow): вкладки те же, но Запуск идёт через `/x/run_flow`, а
  Настройка/Шаблон доступны частично. Быстрее, но раздвоение под капотом остаётся.
- **C. Пока не сливать артефакт**, только UI-вход (обе кнопки под «Создать»), артефакты живут раздельно
  как сейчас. Минимальный риск, слияние артефакта — потом.

**Для первого шага — вариант C** (UI-вход), артефакт-слияние (A) — отдельным крупным инкрементом,
когда UI устоится. Так не трогаем `wz_auto_compose`/`wz_flow_run` раньше времени.

## 4. Экраны

- **«Создать»** (новая вкладка вместо `Composer`; `Мастер` уходит внутрь) — ДВЕ РАВНОПРАВНЫЕ КАРТОЧКИ:
  - Заголовок: «Что хотите поручить Extella?».
  - Карточка 1 «⚡ Собрать задачу из готовых блоков»: поле-фраза + «Собрать» (`composeNow`) + примеры-чипы.
    Подпись: разбор файлов, дайджесты, поиск, отчёты — из блоков библиотеки, сразу запустит.
  - Карточка 2 «🏢 Спроектировать бизнес-процесс»: кнопка «Пройти интервью с Мастером →» (`newSession`).
    Подпись: процесс, живущий в компании постоянно — данные систем, расписание, получатели, агент 24/7.
  - НЕ оформлять Мастера ссылкой-fallback («если не собралось») — равный визуальный вес.
- **Результат компоновки** (есть): собранные шаги, установки, `missing`.
  - Если `missing` пуст: «▶ Запустить сейчас» (есть) + «＋ Сохранить как автоматизацию» (в кабину).
  - Если `missing` непуст: рядом с «🌐 Найти в интернете» — **«🧩 Достроить Мастером»** (НОВОЕ):
    открывает Мастер, преднаполненный задачей и списком пробелов; после стройки эксперта — «Пересобрать».
- **Мастер** — остаётся как есть, но:
  - вызывается из «Создать» (кнопка/ссылка), не отдельной вкладкой навигации;
  - принимает необязательный контекст `{task, gaps[]}` для режима «достроить пробел».

## 5. Порядок сборки (инкременты)

1. **UI-вход «Создать»** (вариант C): переименовать `Composer`→`Создать`, убрать `Мастер` из
   верхней навигации и дать вход в него ссылкой «или пройти интервью». Артефакты не трогаем.
2. **Кнопка «🧩 Достроить Мастером»** на экране компоновки при непустом `missing`: открывает Мастер с
   преднаполнением `{task, gaps}`. (Мост: пробросить контекст; Мастер: показать пробел первым вопросом.)
3. **«＋ Сохранить как автоматизацию»** для composed-flow → минимальная запись, чтобы flow появлялся в
   списке «Мои автоматизации» и открывался в кабине (хотя бы Запуск/Прогоны). Вариант B-lite.
4. **Слияние артефакта (A)**: `wz_auto_compose` → процесс-сессия с flow-оркестратором; composed-flow
   получает полную кабину (Настройка/Шаблон/расписание/получатели). Крупный инкремент, отдельно.
5. **Роутинг по намерению** (не только по пробелу): Композитор, получив фразу с признаками
   бизнес-процесса (данные компании + регулярность + получатели/ответственные — «дебиторка 28 филиалов
   каждую неделю директору»), сам говорит: «Это похоже на бизнес-процесс — предлагаю интервью с
   Мастером, там надёжнее». Классификация — Qwen (задача vs процесс) до/вместе с компоновкой.
   Плюс обратное: Мастер сначала спрашивает Композитор о реюзе. Система понимает разницу
   «задача/процесс» — пользователю не нужно её знать. Финальная «умная» склейка.

## 6. Открытые вопросы

- Composed-flow доставка/расписание: `wz_flow_run` умеет `delivered`, но получатели/шаблон §Кабина —
  пока только у built. Решается в инкременте 3–4.
- «Достроить Мастером» кодогенит эксперт под пробел — нужен чистый интерфейс «пробел → мини-ТЗ →
  один эксперт → в библиотеку», не полный процесс. Возможно, отдельный лёгкий build-путь.
- Библиотеко-осознанность Мастера: он уже реюзит через `wz_build_plan`; проверить, что не строит заново
  то, что Композитор уже нашёл.

## 7. Инварианты (канон)

- Весь LLM — только Qwen (компоновка, кодоген, интервью), никогда Claude.
- Правки/решения — в сессию (`~/extella_wizard/sessions/`). Продовые агенты заморожены (F2).
- Скоупинг per-agent; общее — `global: true`.
- Не трогать `wz_auto_compose`/`wz_flow_run` на платформе, пока UI-вход (инкремент 1–2) не устоится.
