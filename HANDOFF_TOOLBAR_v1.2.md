# Handoff тулбару — v1.2 (кабинет агента, личности, живые базы знаний)

Эти изменения сделаны в Визарде (этот репо). В тулбаре (`plugins_manager.html`, отдельный репозиторий, другой владелец) нужна интеграция потребляющей стороны. Ниже — контракт и что интегрировать.

## 1. Контракт карточки агента (что Визард пишет в витрину)

Визард при `/x/publish` кладёт карточку в KV-каталог **`_mkt_automations`** (глобальный, читать/писать под `agent_extella_default`). Схема карточки (новые поля жирным):

```jsonc
{
  "id": "extella-<session>",
  "name": "Название процесса",
  "type": "process",
  "description": "…",
  "experts": ["<ns>_stage1", "<ns>_search_...", "<ns>_run_pipeline"],
  "orchestrator": "<ns>_run_pipeline",
  "source": "https://github.com/…" | "local",
  // ── v1.2: индивидуальная личность (генерит Qwen) ──
  "emoji": "📋",                 // по роли
  "accent": "#2E6DA4",           // hex-цвет обложки
  "tagline": "Кадровый приказ за минуты, без ошибок",
  "capabilities": ["Черновики приказов", "Ссылки на статьи ТК РК", …],
  "category": "Документы|Продажи и клиенты|Контент|Автоматизация",
  "status": "active|paused|frozen",
  "sessionId": "wz_…"            // связь с планировщиком (/x/schedule)
}
```

Все поля кроме первых — опциональны: если их нет (старые карточки), тулбар выводит фолбэк (эмодзи ⚙️, цвет по хешу id, умения из имён под-экспертов).

## 2. Что интегрировать в тулбаре

Во вкладке **«AI Автоматизации»** каждый агент — управляемый объект, не плоская строка:

- **Hero-карточка**: обложка = градиент из `accent` + крупный `emoji` + белый заголовок + `tagline`; ниже — чипы `capabilities`, статус-бейдж, кнопки «⚙ Кабинет» и «▶ Запустить».
- **Кабинет агента** (полноэкранная панель, 4 вкладки):
  - **Обзор** — описание, умения-чипы, «из чего собран» (список `experts` — Lego-композиция).
  - **Запуски** — история из KV `agent_runs:<id>` (пишет эксперт `wz_agent_runlog`, action `append`/`list`); кнопка «Запустить сейчас».
  - **Расписание** — вызывает мост Визарда `POST http://127.0.0.1:8765/x/schedule` `{session_id, period, interval_min, deliver}` (нужен `sessionId` из карточки); пресеты час/6ч/день/неделя.
  - **Настройки** — переименовать, пауза/заморозка, «изменить через Строителя», убрать из витрины. Изменяемое состояние — клиентский слой KV `agent_state:<id>` (`{name?, status?, schedule?}`), накладывается поверх карточки.

## 3. Важные детали интеграции

- **Скоуп KV**: витрину `_mkt_automations` и `agent_runs:*` читать под тем же агентом, под которым пишет Визард (`agent_extella_default`) — иначе «агент-тень» (карточка/история не видны). Урок дорого дался (см. CHANGELOG 1.2.0).
- **Запуск агента**: `etb_run_expert <orchestrator>`; HTTP 500/timeout ≠ провал (канон) — помечать «в фоне», не «ошибка».
- **Расписание**: реальный крон живёт в Визарде (`sched:<sid>` + тик на VPS 24/7). Тулбар только вызывает `/x/schedule`; сам крон не держит.

## 4. Живые базы знаний — тулбару делать НЕ надо

Knowledge grounding (агент запрашивает кодекс через `kp_ask`) полностью на стороне Визарда/сборки. Тулбару достаточно показывать под-эксперт `<ns>_search_...` в «из чего собран» — он и есть knowledge-стадия.
